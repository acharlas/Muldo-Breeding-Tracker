Architecture — State & Data Fetching

How do components get their data from the API?

A

Zustand store per domain (recommended)

One store per view: useCascadeStore, useInventoryStore, usePlannerStore, useHistoriqueStore. Each store owns its data + loading state + fetch action. Planner store uses Zustand's built-in persist middleware for localStorage.

// cascade.ts
useCascadeStore = { items, loading, fetch() }

// planner.ts — persisted to localStorage
usePlannerStore = { plan, results, generate(), setResult(), submit() }

// Component usage
const { items, fetch } = useCascadeStore()
useEffect(() => fetch(), [])

Pour

  • Zustand déjà installé
  • Persist middleware intégré pour localStorage
  • Simple et prévisible
  • Données partagées entre composants sans prop drilling

Contre

  • Pas de cache automatique (refetch manuel)
  • Pas de déduplication des requêtes
B

TanStack Query

Bibliothèque de server-state dédiée. Cache automatique, refetch en arrière-plan, déduplication. Ajoute une dépendance mais c'est l'outil standard pour ce type de cas.

const { data, isLoading } = useQuery({
  queryKey: ['cascade'],
  queryFn: () => api.get('/api/cascade'),
})

// Mutation pour batch breed
const mutation = useMutation({ mutationFn: ... })

Pour

  • Cache & refetch automatiques
  • Gestion loading/error intégrée
  • Invalidation de cache après mutation

Contre

  • Nouvelle dépendance à installer
  • Overkill pour un tracker usage unique
  • localStorage du plan à gérer séparément quand même
C

useState + useEffect par composant

Chaque vue gère son propre état avec des hooks React basiques. Pas de store global. Simple au démarrage mais limité quand les vues partagent des données (ex: cascade affectée après un breed).

const [items, setItems] = useState([])
const [loading, setLoading] = useState(true)

useEffect(() => {
  api.get('/api/cascade').then(setItems)
    .finally(() => setLoading(false))
}, [])

Pour

  • Aucune dépendance supplémentaire
  • Très simple à comprendre

Contre

  • Données perdues au changement de vue
  • Prop drilling si besoin de partager
  • localStorage à gérer manuellement
